2. 观测空间诊断

2.1 🔴 高严重性 —— 坐标参考系混用

问题：self_state 的位置用的是 绝对竞技场坐标（归一化到 arena_half），而所有实体的位置是 以自我为中心的相对坐标（归一化到 vision_radius）。

这意味着网络需要同时理解两套不同的坐标参考系，增大了学习难度。在 ego-centric 观测中，自身绝对位置的价值非常有限——因为 map_state 射线已经提供了周围障碍物信息，而球/敌人已经在相对坐标中给出。

理论依据： 在 POMDP 设置下，ego-centric 观测更符合马尔可夫性质（Mnih et al., 2016; OpenAI Five, 2019）。绝对坐标只在需要跨 episode 记忆地图时才必要。

2.2 🔴 高严重性 —— Slot 编码的排列敏感性问题

问题： 当前使用固定 slot + 按距离排序的编码方式。当实体进出视野时，slot 编号会跳变：

# 场景：2个球在视野内 slot_0 = [球A信息] slot_1 = [球B信息] slot_2..7 = [0*] # 球A被吃掉后： slot_0 = [球B信息] slot_1..7 = [0*] # → 球B的信息从 slot_1 跳到了 slot_0！

网络必须额外学习"slot 编号无意义"这一不变性，引入了不必要的归纳偏置噪声。对 Transformer/MLP 这种无排列不变性的架构尤其有害。

标准解决方案：

  • 方案 A：Self-Attention 池化 —— 将所有实体编码为等长向量后，用 multi-head attention 做加权聚合（类似 Set Transformer, Lee et al. 2019）
  • 方案 B：网格化（Grid-based） —— 将视野区域划分为网格（如 7×7），每个网格单元编码该位置是否有实体/实体属性，天然具有空间排列不变性
  • 方案 C（最小改动）： 使用 Sum/Max Pooling 替代 Slot —— 将所有视野内实体编码后做 sum/max 降维，丢失部分信息但消除排列敏感性

2.3 🟡 中严重性 —— 缺失速度与时间信息

问题： 观测中完全没有自身速度、自身动作历史、或其他实体速度。在连续动作环境中（虽然是离散动作），速度是关键的动态信息。没有速度，智能体无法：

  • 判断自己是否在移动（需要从连续帧的 pos 变化推断）
  • 预测其他实体的轨迹（躲避敌人）
  • 做出精确的到达判断

建议： 自身速度添加到 self_state；对其他玩家/敌人添加速度向量。如果担心维度膨胀，可只添加自身速度。

2.4 🟡 中严重性 —— 冗余特征与不必要特征

冗余： 实体同时编码了 rel_x, rel_y 和 dist/vision_radius。由于 dist = sqrt(rel_x²+rel_y²)*vision_radius，这是完全确定的冗余。不如用 (rel_x, rel_y) 或 (dist, cos_theta, sin_theta) 二选一。

不必要（对"避障吃球"任务）：

  • flip_h —— 只影响攻击方向，吃球任务不需要
  • enemy hp_ratio —— 如果策略是"避障吃球"而非"战斗"，敌人血量无关
  • other_player hp_ratio —— 同上，除非考虑 PvP 吃球竞争

2.5 🟡 中严重性 —— 缺失帧堆叠/记忆

问题： 当前是单帧观测。对 POMDP 性质的游戏（部分可观测 + 需要推断速度/轨迹），帧堆叠（frame stacking, 通常 3-4 帧）或 LSTM 是标准做法（Mnih et al., 2015, DQN Nature paper）。

影响： 没有帧堆叠时，智能体需要从 ray distance 的逐帧变化间接推断碰撞方向，这非常困难。

2.6 🟢 低严重性 —— 射线分辨率与表示

32 条射线在 200px 视野半径下，角分辨率 11.25°，像素分辨率 ~39px（周长方向）。对于 960px 的竞技场，200px 半径覆盖了约 13.6% 面积。这个配置在 定性上是合理的，但在狭窄通道处可能不够。

小优化： 考虑增加到 48 条（7.5°分辨率）或使用 CNN 处理 32 条射线的局部相关结构（类似 1D 卷积）。

2.7 🟢 低严重性 —— 目标信息不完整

球类型用 0/1 二进制标记，但智能体不知道 A 球不重生、B 球会重生。如果目标是最优收集策略，这个信息很重要：A 球应该优先吃（一次性的），B 球可以稍后吃。

2.8 🟢 低严重性 —— 饥饿时间未入观测

代码中被注释掉了（play_scene.gd:536-537）。饥饿是奖励函数的组成部分，如果智能体不知道自己的饥饿状态，就无法做出最优决策来避免饥饿惩罚。

3. 奖励设计诊断

3.1 🔴 高严重性 —— 目标冲突：吃球 vs 战斗

核心问题： 当前奖励同时激励吃球和战斗。但击杀玩家 (+45) 远大于吃球 (+10/+15)。这意味着：

  • 最优策略可能是 忽略球，优先击杀
  • 吃球只是"顺便的事"，不是主要学习信号
  • 在"避障吃球"的论文叙事下，战斗奖励是 confounding factor（混杂因素）

解决方向： 明确实验边界。如果论文定位是"避障吃球"任务，则应该移除 PvP 战斗奖励（或作为消融实验的变体）。战斗系统可以作为独立任务或扩展来评估。

3.2 🔴 高严重性 —— 势能塑形的数值不稳定风险

问题： 球势能塑形使用 scale=1.0, γ=0.99，Φ 的最大值等于球奖励值（10 或 15）。

当 γ≈1 且 scale=1 时，塑形奖励 ≈ 稀疏奖励值本身：

# 假设 agent 从视野边缘跑到球旁边： Φ_start = 1.0 × 10 × exp(-5×200/200) = 0.067 (在边缘) Φ_end = 1.0 × 10 × exp(-5×0/200) = 10 (在球旁) # Ng et al. 塑形： F = 0.99 × 10 - 0.067 = 9.833 （接近吃球的10点奖励！）

这意味着：塑形奖励的累计值可能超过实际吃球奖励，智能体可能学会"靠近球但不吃"来持续获取塑形奖励。虽然 Ng et al. 理论保证策略不变性，但在 函数近似 + 有限样本 条件下，过大的塑形奖励会主导梯度方向。

建议： ball_potential_scale 降低到 0.1~0.3，或使用基于距离的归一化（如 Φ = scale × reward × φ(d/vision_radius) 其中 φ 是衰减函数且 φ(0)=φ_max ≪ 1）。

3.3 🟡 中严重性 —— 饥饿机制与执行时长不匹配

问题： 饥饿在 1500ms (1.5s) 后触发，最大惩罚持续 10s。在 200px 视野半径下，agent 从视野边缘跑到中心球需要约 200/100 = 2s（假设 run_speed=100）。这意味着：

  • 如果附近没有球，agent 在到达球之前就已经开始饥饿
  • quadratic 惩罚函数在后期增长很快，可能迫使 agent 采取激进（危险）行为

建议： starve_time 增加到 3000-5000ms，或改用 sqrt 增长函数使初期惩罚更温和、后期趋缓。

3.4 🟡 中严重性 —— 撞墙惩罚过小

-0.05/帧 撞墙惩罚太小。在 60fps 下，连续贴墙 1 秒只有 -3 点，而吃一个球 +10。agent 可以在探索阶段学到"沿墙壁滑行"是可行的导航策略。这 不是 bug（滑墙确实是有效的），但过小的惩罚延迟了 agent 学习主动远离墙壁。

建议： 增加到 -0.1 ~ -0.2/帧，配合 wall_potential_shaping 使用 LINEAR 模式。

3.5 🟡 中严重性 —— 缺失有效的探索引导

问题： center_reward_scale=0（已禁用），run 惩罚=0（已禁用）。agent 在初期没有任何密度奖励引导它离开出生角。四个出生角各有一些 A 球，agent 吃完自己角的球后可能停在那里不动。

建议：

  • 启用 center_shaping（小幅度，如 scale=0.05~0.1），引导 agent 向中心探索
  • 或用少量的 curiosity-driven exploration（Intrinsic Curiosity Module, Pathak et al. 2017）
  • 或用 count-based exploration 惩罚停留区域

3.6 🟢 低严重性 —— 奖励量级跨类别不平衡

稀疏奖励范围（-20 ~ +45）远大于密集奖励（-0.05 ~ +0.25/帧）。虽然有 VecNormalize 归一化，但原始量级的不平衡仍会影响早期训练。建议将稀疏奖励 scale down 或在归一化前做 clipping。

3.7 🟢 低严重性 —— run 惩罚完全禁用

run=-0.0 意味着移动没有代价。消除所有移动惩罚可能导致 agent 学习无意义的往复运动。建议设为 -1e-5 ~ -1e-4 的小值，维持效率压力但不影响主要目标。

4. 重构方案：避障吃球纯化设计

🌟 设计哲学

将任务重新定位为 纯"避障吃球"——让 RL agent 在静态障碍物 + 移动敌人环境中最大化球收集效率。剥离战斗系统对奖励的影响，使论文叙事与实验设计一致。

4.1 重构后的观测空间 64 维

核心设计原则
  1. 统一 ego-centric 坐标系 —— 所有空间信息以 agent 为中心
  2. 消除冗余 —— 只保留 (rel_x, rel_y) 或 (dist, cosθ, sinθ)
  3. 加入速度/时间信息 —— 自身速度 + 球/敌人的距离变化率
  4. 使用 Pooling 替代 Slot —— 消除排列敏感性

4.3 关键设计理由

1. 奖励量级归一化： 将球奖励从 10/15 改为 1.0/1.5，不是降低了重要性，而是让所有奖励在同一量级。配合势能塑形 scale=0.15 后，塑形奖励累计不会超过稀疏奖励。也避免了 VecNormalize 的长期统计漂移问题。

2. 势能塑形衰减函数改为 LINEAR： 当前 EXPONENTIAL 函数 exp(-5*d/r) 中的 5 倍系数使势能在球附近剧烈变化。LINEAR 函数 Φ = scale * reward * (1 - d/vision_radius) 梯度恒定，更容易学习。这通过 reward_config.ball_potential_func 即可切换。

3. 移除战斗奖励： 这不是说战斗系统不好，而是 论文任务需要明确边界。可以在消融实验中对比"纯吃球 vs 吃球+战斗"两种奖励设置的效果。

4. Pooling 替代 Slot： 对于球和敌人的观测，使用 mean/max/min pooling 聚合后，agent 不再需要学习 slot 索引无关性。对于"避障吃球"任务，agent 只需要知道"最近球在哪"和"敌人在哪个大致方向"，不需要精确区分每个实体。

如果不想做大改动，最少改 3 个参数即可验证收益：

总结

当前设计的核心矛盾：

  1. 观测空间混用了两种坐标参考系 + Slot 编码排列敏感，增加了不必要的学习难度
  2. 奖励设计混合了吃球和战斗两个目标，导致信号不纯净
  3. 塑形奖励量级过高（与稀疏奖励相当），在有限样本下可能误导梯度

推荐方向：纯化任务为"避障吃球"，采用 Pooling 观测 + 归一化奖励 + 温和塑形 + 帧堆叠。同时保留战斗系统作为消融实验的对照组，增强论文的实验丰富度。
